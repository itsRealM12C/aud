AU Extractor — Codec Candidate Analyzer

For proprietary .au-style containers with no known codec tag (e.g. CloudPets-style audio blobs). Instead of committing to one decoder, this tries several candidate PCM interpretations of the same payload bytes and ranks each by how speech-like its spectrum looks — dominant energy in the ~250–3400 Hz voice band and non-trivial frame-to-frame activity, rather than a flat/near-DC spectrum.

Why this exists: a raw decode can succeed technically (valid sample values, no crashes) while still being the wrong codec — the audio it produces is just noise shaped like audio. Example from a real session: a GSM 6.10 decode of a payload came back with top spectral components at 1–18 Hz — nowhere near the human voice range — a strong sign that GSM was the wrong codec for that stream, even though the decode itself didn't error out. Scoring every candidate this way catches that automatically instead of eyeballing FFT output.

GSM 06.10 (real ffmpeg, via ffmpeg.wasm)
Runs the actual command ffmpeg -f gsm -ar <rate> -ac 1 -i payload.bin out.wav in-browser — a hand-written GSM 06.10 decoder is easy to get subtly wrong (RPE-LTP math + bit-packing), so this uses the real reference decoder instead of reimplementing it. First click downloads the ffmpeg core (~25 MB, cached after that).
GSM frames are stateful — each 33-byte frame decodes using the previous frame's state, so if the header size guess is off by even a few bytes, every frame decodes from a corrupted mid-frame position and you get exactly the "mostly noise, occasionally speech-like" result. This tries a spread of header offsets through the real decoder and scores each, so you can find the byte-exact boundary instead of guessing.
